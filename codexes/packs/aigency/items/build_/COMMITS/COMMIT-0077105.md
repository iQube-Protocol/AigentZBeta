## Commit Message

```
intent chains commit 9: MyWorkspaceTab clickable intent cards open drawer

Active Intents list items are now buttons. When a chain is attached to
the intent, click opens the ChainDetailDrawer (commit 8) showing the
full step history + cancel/feedback footer. When no chain is attached,
the card stays as a passive display (existing behaviour).

Implementation notes:
- New state: intentToChain map + drawerChainId + drawerOpen
- On tab activation, fetches GET /api/intent-chains?limit=200 once and
  builds a lightweight map. The list endpoint omits chain.context for
  bandwidth, so intent_id → chain_id correlation isn't precise yet —
  v1 falls back to "most recent chain" via a __lastChain key when an
  exact intent match doesn't exist. v1.1 will fetch chain context per
  row for precise correlation (small follow-on item).
- The intent card chip surface shows "↳ chain" hint on hover-able
  rows so the user knows the click is meaningful
- Disabled state preserves the existing read-only behaviour for
  intents with no chain
- Drawer mounts at the tab root level (outside the scrollable content
  area) as a fixed inset overlay so it works on all subtabs

This closes the visible end-to-end loop for the marketa worked example:
1. User clicks CTA "Ask Marketa for a partner proposal"
2. Chain dispatches, brief composer opens (commit 7)
3. User writes brief → artifact created → chain advances to Marketa
   propose RPC step (commit 7 + 5)
4. Marketa emits proposal_drafted, chain advances to review-proposal
5. User goes to myWorkspace tab → sees the intent card with ↳ chain
6. Clicks → ChainDetailDrawer shows the full step history including
   the rendered proposal + can submit feedback when the chain
   terminates
```
