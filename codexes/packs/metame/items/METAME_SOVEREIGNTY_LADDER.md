# metaMe Canonical Ladder Spec

**Status:** canonical  
**Authority:** product owner  
**Last updated:** 2026-04-07

## 1. Purpose

This spec defines the **canonical metaMe sovereignty ladder** for the AgentiQ / metaMe ecosystem.

It is intended to:

* provide a platform-wide canonical progression model
* align with Progressive Creative Sovereignty (PCS)
* support the user's goals, matrix, ladder, and next-best-pathway logic inside the metaMe Cartridge
* allow cartridges such as **KNYT** to apply their own narrative or branded tier skins on top of the same underlying progression model
* support policy, access, onboarding, and contribution pathways consistently across the stack

---

## 2. Canonical ladder

The recommended canonical metaMe ladder is:

1. **Visitor**
2. **Initiate**
3. **Participant**
4. **Curator**
5. **Composer**
6. **Operator**
7. **Architect**

### Canonical interpretation

This ladder tracks the user's progression from:

* awareness
* activation
* participation
* curation
* creation
* stewardship / operation
* upstream ecosystem contribution

### Canonical shorthand

**Visitor → Initiate → Participant → Curator → Composer → Operator → Architect**

---

## 3. Why this ladder

This canonical ladder is designed to:

* stay **general** across cartridges and domains
* remain aligned with **PCS**
* clearly support the progression from user to contributor
* function as a **policy and access model**, not just a branding layer
* support cartridge-specific narrative skins without fragmenting the platform's underlying logic

## 6. Tier definitions

## 6.1 Visitor

### Meaning

A person who is present at the edge of the ecosystem but has not yet fully activated into it.

### Typical state

* browsing
* previewing experiences
* not yet fully onboarded
* may not yet have activated a persona/wallet/profile surface

### Role in PCS

Pre-entry state. This is before meaningful sovereignty participation begins.

### Typical permissions

* view public-facing surfaces
* preview selected cartridge experiences
* view introductory guidance
* receive onboarding prompts

### Typical next-best actions

* create or activate persona
* connect wallet or identity layer if needed
* enter a starter flow
* complete onboarding

### metaMe significance

This is the outermost surface of the sovereignty journey.

---

## 6.2 Initiate

### Meaning

A person who has activated into the ecosystem and completed basic onboarding.

### Typical state

* persona activated
* identity/wallet or equivalent entry requirements satisfied
* basic orientation complete
* first meaningful access granted

### Role in PCS

Beginning of explicit sovereignty participation.

### Typical permissions

* access basic Runtime experiences
* save personal preferences/state
* receive guided next-best experiences
* enter beginner participation flows

### Typical next-best actions

* engage in first cartridge experience
* complete first participation action
* join a community or theme pathway
* begin signaling preferences/interests

### metaMe significance

This is the first stable identity-aware state inside the metaMe sovereignty layer.

---

## 6.3 Participant

### Meaning

A person actively engaging with Runtime and generating downstream signal.

### Typical state

* interacts repeatedly
* signals preferences
* engages in community activity
* consumes and participates meaningfully

### Role in PCS

This is the first clearly productive signal-generating tier.

### Typical permissions

* vote
* like
* spark
* collect
* participate in basic challenges, flows, or quests
* receive reward eligibility where appropriate

### Typical next-best actions

* deepen activity in chosen cartridge(s)
* begin remix-capable flows if eligible
* contribute more consistent participation signals
* identify preferred pathways toward curation or creation

### metaMe significance

This tier is the foundation of Runtime signal.

---

## 6.4 Curator

### Meaning

A person who not only participates, but helps shape, organize, or elevate signal and community-facing value.

### Typical state

* remixes selectively
* curates content or pathways
* surfaces signal that matters
* acts as a meaningful community or correspondent presence

### Role in PCS

This is the first tier where a user meaningfully influences the shape of the ecosystem experience for others.

### Typical permissions

* remix selected content or flows
* correspond or contribute in more visible ways
* curate collections, pathways, or recommendations
* influence ranking, discovery, or signal surfaces where allowed

### Typical next-best actions

* create structured remixes
* surface insights or summaries
* move into guided composition
* begin preparing creator/operator pathways

### metaMe significance

This tier begins the transition from consumption toward stewardship and creation.

---

## 6.5 Composer

### Meaning

A person who can assemble, create, or materially shape new experiences.

### Typical state

* uses Studio or analogous creation surfaces
* creates derivative or net-new experience flows
* composes guided outputs, capsules, pathways, or structured experiences

### Role in PCS

This is the first clearly creator-grade tier.

### Typical permissions

* access creation or composition tooling
* build experiences, capsules, or guided flows
* publish or stage selected outputs
* contribute cartridge-level experiential value

### Typical next-best actions

* deepen Studio usage
* produce higher-quality compositions
* move toward more governed contribution
* support community and cartridge operations where relevant

### metaMe significance

This is where the user becomes an active creator in the sovereignty cycle.

---

## 6.6 Operator

### Meaning

A person who can steward, manage, or run governed parts of the ecosystem.

### Typical state

* manages flows or spaces
* stewards cartridge/codex surfaces
* governs permissions or contribution processes
* supports programmatic or operational execution

### Role in PCS

This tier marks the transition from creator to steward/operator.

### Typical permissions

* manage governed spaces
* moderate or steward cartridge surfaces
* operate higher-trust workflows
* submit or manage structured contributions
* interact with more sensitive system layers depending on policy

### Typical next-best actions

* take responsibility for a zone, program, or service
* support Registry-bound contribution pathways
* help govern promotion, access, or execution surfaces
* move toward upstream/system contribution

### metaMe significance

This tier is where sovereignty includes operational stewardship.

---

## 6.7 Architect

### Meaning

A person who contributes at the upstream ecosystem or structural level.

### Typical state

* contributes patterns, harnesses, services, structures, or systems
* shapes the ecosystem beyond a single experience flow
* acts as a founding, apex, or upstream builder/steward layer

### Role in PCS

This is the highest canonical expression of ascension: the user becomes an upstream contributor.

### Typical permissions

* contribute to Registry/OS-level supply
* shape ecosystem patterns and structures
* define or influence high-level frameworks and governance in approved ways
* operate at the highest trust and contribution layer available to that user type

### Typical next-best actions

* upstream contribution to AgentiQ OS / Registry / shared patterns
* system-level stewardship
* domain architecture or cartridge-framework influence
* mentoring lower tiers into contribution pathways

### metaMe significance

This is the canonical fulfillment of PCS.

---

## 7. Permissions model

The ladder should not be treated as branding only. It should drive **policy and access**.

### Principle

Each tier should control some combination of:

* what the user can see
* what the user can do
* what the user can influence
* what the user can create
* what the user can steward
* what the user can contribute upstream

### Permission families

Recommended permission families:

* `view`
* `participate`
* `signal`
* `remix`
* `curate`
* `compose`
* `publish`
* `steward`
* `operate`
* `promote`
* `contribute_upstream`

### Ladder implication

Not every cartridge needs to use all permission families, but the canonical ladder should map to them consistently.

---

## 8. Unlock logic

Tier movement should not be purely cosmetic.

### Unlock types

A new tier may unlock:

* new experience categories
* new responsibilities
* new contribution actions
* new trust scopes
* new cartridge surfaces
* new codex interactions
* new reward opportunities
* new next-best-pathway recommendations

### Principle

Unlocks should be based on:

* demonstrated activity
* consistency
* signal quality
* goal alignment
* contribution readiness
* stewardship readiness
* policy constraints

## 9. Next-best-action logic by tier

metaMe should use the ladder as a core input for next-best-pathway logic.

### Visitor

Next best action: activate.

### Initiate

Next best action: participate for the first time.

### Participant

Next best action: deepen participation and begin shaping signal.

### Curator

Next best action: move from shaping signal to composing value.

### Composer

Next best action: move from creation to stewardship or contribution.

### Operator

Next best action: move from stewardship to upstream ecosystem contribution.

### Architect

Next best action: deepen upstream contribution and support the ascension of others.

## 13. Suggested progression criteria

These should remain flexible, but the following logic is a good starting point.

### Visitor → Initiate

* complete onboarding
* activate persona / identity surface
* enter first guided experience

### Initiate → Participant

* perform first meaningful participation actions
* show repeated engagement
* complete starter pathway

### Participant → Curator

* demonstrate consistent signal
* engage in community/correspondent activity
* perform structured remix or curation actions

### Curator → Composer

* create or assemble meaningful derivative outputs
* use composition or creation tools in a more structured way
* demonstrate ability to add experiential value

### Composer → Operator

* steward flows or spaces
* manage governed surfaces
* support higher-trust activities
* demonstrate reliability and judgment

### Operator → Architect

* contribute upstream patterns, services, structures, or assets
* demonstrate ecosystem-level value
* operate with high trust and alignment

## 17. Experience matrix position

The sovereignty ladder provides the **X-axis** of the experience matrix:

```
                        Sovereignty journey (X) →
                 Visitor  Initiate  Participant  Curator  Composer  Operator  Architect
Steward    ↑   [  ...      ...        ...         ...       ...       ...       ★    ]
Builder    |   [  ...      ...        ...         ...       ...       ★         ...  ]
Producer   |   [  ...      ...        ...         ...       ★         ...       ...  ]
Modifier   Y   [  ...      ...        ...         ★         ...       ...       ...  ]
Selector   |   [  ...      ...        ★           ...       ...       ...       ...  ]
Recipient  |   [  ★        ...        ...         ...       ...       ...       ...  ]
```

- **Y-axis** = PCS engagement level (abstract canonical: Recipient → Steward)
- **X-axis** = Sovereignty journey (this ladder: Visitor → Architect)
- **Cell** = NBE prescription: format (pill/capsule/mini_runtime/codex) + intent
- **Goal** = move users toward the **top-right** quadrant
- **Pathways** = many possible routes to top-right; dynamic per cohort, individual, and signals

### Lenses on the matrix

| Lens | Shows | Use case |
|---|---|---|
| **Individual** | One person's (Y,X) position + their NBE pathway | Aigent C serving the individual |
| **Cohort** | Density heatmap — where are users clustered? Where stalled? | Cohort analysis, stall detection |
| **Organizational** | Full population distribution, funnel health, strategic interventions | Marketa campaign planning |
