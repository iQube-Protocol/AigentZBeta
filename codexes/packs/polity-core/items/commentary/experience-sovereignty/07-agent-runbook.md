1. Purpose
This runbook helps agents assist users with Experience Vibing, iQube preparation, Cartridge
creation, and publishing-route decisions across the metaMe stack.
It is written for agents that support users, creators, operators, developers, and organizations working
with:
-metaMe Runtime
-metaMe Studio
-iQube Registry
-nanOS
-AgentiQ OS
-myCanvas
-Cartridges
-Codex-grade assets
-Capsules, Pills, Chips, Buttons, and Badges
-Studio Composer
-aigentMe
-Cartridge CoPilots
-metaMe Experience Catalogue
The agent's job is to reduce user cognitive load while preserving user agency.
The user provides:
-domain expertise
-desired outcome

-Registry preparation
-Runtime activation packages
-mini-RFPs for missing workflows or skills
The agent recommends. The user authorizes. The metaMe stack validates, filters, and activates.

3. System Map for Agents
Agents should understand the corrected operating stack as:
AgentiQ OS
-> iQube Registry
-> nanOS
-> metaMe Studio
-> metaMe Experience Catalogue
-> metaMe Runtime / User Runtime Activation
AgentiQ OS is the open-source creation and interoperability layer. It supports complex iQube
primitives, tools, workflows, agents, integrations, and cartridge-compatible modules.
iQube Registry is the canonical index and first production-grade filter for iQube primitives. Anything
entering the metaMe ecosystem must pass the Registry hurdle.
nanOS sits between the iQube Registry and metaMe Studio. It filters Registry assets for Studio-grade
and Runtime-grade use and operationalizes iQubes, Personas, Cartridges, SmartTriad, Smart
Content, Liquid UI, Smart Wallets, and policy-governed interaction.
metaMe Studio is the professional-grade composition layer. It is where experiences are composed,
packaged, previewed, reviewed, and prepared for Catalogue approval or Runtime use.
metaMe Experience Catalogue is the controlled distribution surface for approved Runtime-ready
experiences. It is like an App Store for sovereign experiences, with policy review, quality controls,
provenance requirements, and consumer protections.
metaMe Runtime is the user's sovereign experience environment. It is where users activate
Catalogue-approved or private experiences, activate Cartridges, interact with Capsules and Pills,
receive Next Best Experience recommendations, and govern experiences through aigentMe.

4. Key Concepts Agents Must Understand
Runtime: the user's live sovereign experience space. It can surface Cartridges, Capsules, Pills,
Codex-grade assets, agents, workflows, Smart Wallet actions, and recommendations.
Cartridge: a directed experience domain. It may represent a company, storyworld, community,
service, venture, project, or personal operating domain. Examples include the metaMe Cartridge,
KNYT Cartridge, Qriptopian Cartridge, Qripto Cartridge, AgentiQ OS Cartridge, Marketa Cartridge,
and metaMe Venture Lab Cartridge.
Codex: both a surface and a grade. A Codex surface is the canonical ledger or knowledge surface
inside a Cartridge. Codex-grade content is canonical-grade content. It may be surfaced outside a
Codex tab while remaining part of the Cartridge's canonical ledger.
Tab: a structured operating surface inside Runtime or a Cartridge. A Tab may function as a page,
drawer, dashboard, quest board, review surface, publishing surface, or active workspace.
Active Surface: where a user does something meaningful: submits, claims, activates, configures,
publishes, approves, joins, completes, or invites.
Capsule: a compound experience unit. It may include content, media, tasks, forms, agent calls, policy
prompts, wallet actions, badges, and receipts.
Pill: a compact experience unit, usually designed for a smaller action or nudge.
Chip: a compressed state, tag, status, or filter object.
Button: a direct action trigger.
Badge: a signal of status, trust, eligibility, achievement, provenance, or progression.
Experience Model: a core instrument of experience sovereignty configured by the user and aigentMe
in the metaMe Cartridge. It consists of Experience Strategy, Experience Ladder, Experience Matrix,
Experience Journey, and Experience State.
aigentMe: the user's personal guide, chief of staff, and Runtime regent inside metaMe. It governs the
user's Runtime relationship, Experience Model, permissions, activations, and cross-cartridge
priorities.
SmartTriad: CoPilot + Codex + Wallet. SmartTriad relationships are typically defined within a
Cartridge and follow the Cartridge's assets wherever they are surfaced. In alpha release, this
relationship is primarily confined to activated Cartridges to reduce complexity.
iQube Card: the agent-readable calling card for an iQube. The Registry is the canonical source; the
iQube Card makes the primitive legible to agents without exposing private payloads or bypassing
policy.

7. Agent Decision Tree
Use this decision tree whenever a user asks to create, remix, publish, activate, or prepare an
experience.
1.Is the user only drafting, exploring, or ideating? -> Prepare a private draft in myCanvas or private
Runtime.
2.Does the experience affect the user's Runtime, goals, identity, priorities, or permissions? -> Hand off
to aigentMe for Experience Model and Runtime policy alignment.
3.Does the experience belong to an existing Cartridge? -> Apply that Cartridge's policy and involve the
Cartridge CoPilot if needed.
4.Does this experience require its own Cartridge? -> Prepare a Cartridge Creation Brief and recommend
myCanvas, Studio, or Registry path.
5.Does this experience require its own Tab within an existing Cartridge? -> Prepare a Tab Creation
Brief and submit under that Cartridge's policy.
6.Is the experience a contribution to a third-party Cartridge? -> Prepare a Cartridge Contribution Brief
and route to the Cartridge CoPilot or review surface.
7.Is the experience canonical, official, or intended to become canonical? -> Prepare for Codex-grade
review and possible Registry submission.
8.Does the experience require a new iQube primitive? -> Prepare an iQube Preparation Brief and hand
off to the Registry workflow.
9.Does the experience require professional composition? -> Hand off to Studio Composer.
10.Does the experience require a missing skill, workflow, agent, or integration? -> Ask Studio Composer
to generate a mini-RFP.
11.Is the experience intended for private user use only? -> Publish to myCanvas or private Runtime.
12.Is the experience intended for limited testing? -> Prepare tester invitation and limited-access policy.
13.Is the experience intended for third-party social publishing? -> Prepare social export, respecting
source Cartridge policy and user rights.

8. Handoff Map
Hand off to aigentMe when the experience affects the user's Runtime, Experience Model, priorities,
policies, identity, or agent permissions. Examples: add this to my daily Runtime; make this part of my
operating model; should this be private or shared; does this align with my goals; which agent should
be allowed to help?
Hand off to a Cartridge CoPilot when the experience belongs to a specific Cartridge domain.
Examples: KNYT experience -> KNYT CoPilot / KnOw1 context; Qriptopian article -> Qriptopian /
Quill context; Qripto experience -> Qripto Cartridge; metaMe Venture Lab workflow -> metaMe
Venture Lab Cartridge; Marketa campaign -> Marketa Cartridge.
Hand off to the iQube Registry when the asset needs canonical primitive status or agent-legible iQube
registration. Examples: ContentQube submission, DataQube creation, ToolQube registration,
ModelQube registration, AigentQube registration, ExperienceQube registration, Cartridge manifest
registration, Codex-grade asset registration, iQube Card generation.
Hand off to nanOS when Registry assets need to be filtered or operationalized for Studio-grade or
Runtime-grade use. nanOS handles Studio compatibility, Runtime compatibility, SmartTriad binding,
Smart Content behavior, Smart Wallet gating, Liquid UI rendering, policy-governed surface use, and
persona/iQube operational alignment. The user agent should not simulate nanOS.
Hand off to Studio Composer when the experience requires professional composition, multi-surface
packaging, or workflow creation. Examples: turn this into a Capsule sequence; create a Cartridge
onboarding flow; package this article into Pills, Capsules, and social prompts; prepare this for
Studio-grade review; generate a mini-RFP for a missing workflow.
Hand off to Runtime when the experience is ready for private activation, limited testing, or
user-facing activation after Catalogue approval where required. Runtime is where the experience is
lived.

10. Cartridge and Tab Decision Guidance
Recommend a new Cartridge when the experience requires its own domain. A new Cartridge may be
appropriate when the experience has its own audience, policies, Codex, CoPilot or agent context,
Smart Wallet logic, contribution rules, publishing workflows, active surfaces, recurring experience
journey, or community/project/venture/service identity.
Examples: a new client experience domain; a new media franchise experience; a private family
operating domain; a professional services operating domain; a partner onboarding environment; a
venture operating Cartridge; a course or learning environment.
Recommend a new Tab when the experience belongs inside an existing Cartridge but needs a distinct
operating surface. A new Tab may be appropriate when the experience belongs to an existing domain,
uses that Cartridge's policies, relies on that Cartridge's Codex, should be navigable inside the
Cartridge, has a repeated function, needs its own workflow surface, and does not require a full
standalone Cartridge.
Examples: Quest Tab inside KNYT; Publishing Tab inside Qriptopian; Partner Review Tab inside
Marketa; Portfolio Dashboard Tab inside metaMe Venture Lab; Runtime Policy Tab inside metaMe;
Codex Review Tab inside a Cartridge.
Recommend a Capsule instead when the experience is a self-contained module but does not need its
own domain or persistent Tab. Examples: onboarding sequence, campaign module, contribution
package, article activation experience, learning module, quest, proposal package.
Recommend a Pill instead when the experience is small, focused, and action-oriented. Examples:
approve, claim, review, read, invite, complete, activate.
11. Publishing Routes
Private myCanvas route: Use when the user wants to draft, review, or privately activate an
experience.
Create -> myCanvas -> private review -> optional private Runtime activation -> optional tester invitation.
Runtime creation route: Use when the user creates directly inside their Runtime.
Create in Runtime -> publish to myCanvas -> activate privately -> invite testers -> submit to Cartridge, Registry,
Studio, Catalogue, or Runtime review if needed.
Cartridge contribution route: Use when the experience belongs to an existing Cartridge.
Create or remix -> identify Cartridge -> apply Cartridge policy -> prepare contribution brief -> submit to Cartridge
CoPilot or review surface -> possible Codex-grade review -> possible Registry / Studio / Catalogue / Runtime path.
Example: A user remixes a KNYT experience in open Runtime and submits it to the KNYT Cartridge
as a community contribution.
Third-party social route: Use when the user wants to publish sovereign content outside metaMe.
Create -> confirm user rights and Cartridge policy -> prepare social export -> publish to third-party platform.
Ecosystem publication route: Use when the asset should enter the general metaMe ecosystem.
Creation -> iQube Registry -> nanOS filtering -> Studio review, if needed -> metaMe Experience Catalogue approval ->
metaMe Runtime activation.

12. iQube Preparation Guidance
The agent helps prepare iQube requirements without asking the user to think technically.
Ask the user for:
-What is the asset or experience?
-Who created it?
-Who is it for?
-What should it help someone do?
-Should it remain private?
-Should metadata be public?
-Should access be token-gated or payment-gated?
-Should it be remixable?
-Should it become canonical?
-Which Cartridge does it belong to?
-Should it be used by agents?
-What actions should be allowed?
Then prepare an iQube brief. The agent may recommend a likely primitive type, but the Registry
validates the final classification.
Likely primitive types include DataQube, ContentQube, ToolQube, ModelQube, AigentQube,
ExperienceQube, CartridgeQube, PolicyQube, and DiDQube / UIDQube.
13. Mini-RFP Guidance
If the needed skill, workflow, integration, or agent capability does not exist, the agent should prepare
a mini-RFP for Studio Composer.
The mini-RFP should protect confidential details and include:
-desired outcome
-missing capability
-target surface
-required inputs
-required outputs
-privacy constraints
-excluded confidential information
-IP rights requirements
-acceptance criteria
-testing criteria
-likely iQube dependencies
-intended publishing path
The user agent should not expose private client details unless the user explicitly authorizes
disclosure.

Addendum A - Corrected Catalogue / Runtime Ordering
In all agent-facing diagrams, decision trees, and publishing-route references, use the following order:
Creation -> iQube Registry -> nanOS filtering -> Studio review, if needed -> metaMe Experience Catalogue approval ->
metaMe Runtime activation.
Operational clarification: The metaMe Experience Catalogue is the controlled distribution surface for
approved experiences. The metaMe Runtime is the user's final activation and execution environment.
Agents should therefore treat Catalogue approval and Runtime activation as distinct steps:
catalogue_status: not_submitted | submitted | approved | rejected | deprecated
runtime_activation_status: not_activated | private_active | tester_active | user_active | suspended
Updated agent routing rule: If an experience is intended for general metaMe ecosystem consumption,
route it through Registry -> nanOS -> Studio review, if needed -> Experience Catalogue -> Runtime
activation.
Do not describe the Experience Catalogue as downstream of Runtime. Runtime is where
Catalogue-approved experiences are activated and lived by the user.
