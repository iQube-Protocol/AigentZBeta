# CFS-020 — The Dynamic Constitutional Interaction Runtime (DCIR)

**Chrysalis Foundation Specification · v1.0 · D0 (contract) — authored 2026-07-06 per the operator's DCIR spec (2026-07-06)**
Constitutional anchor: `codexes/packs/polity-core/constitutional-records/invariant-intelligence.md`
Companion to: CFS-015 (Operation Chrysalis 2.0), CFS-017 (the observe-first seam precedent), CFS-019 (the CCRL — behavioural-invariant ratification lives under its governance).
Invariants: `inv.interaction.112`–`118` (Appendix A). Contract: `types/dcir.ts`.
DCIR stands alongside **Constitutional Reasoning**, **Constitutional Order**, and **Constitutional Action** as a canonical runtime capability.

---

## 1. Purpose

DCIR is the **interaction substrate** of Constitutional Computing: a closed bidirectional cognitive-action loop that continuously synchronizes human intent, inference, application state, generated artefacts, recommendations, and next actions into **one evolving constitutional state**. Its founding refusal: **generation is never terminal — it is a state transition.** Today an artefact is produced and forgotten by the system that produced it; under DCIR, generate a PDF and the PDF **is** state — "make page three shorter" already knows which document, which version, which section (`inv.interaction.114`).

## 2. The closed loop

```
Conversation → Inference → Action → Observation → Updated Context → Inference → …
```

…continuously, until the constitutional objective is satisfied (`inv.interaction.118`). No single generation closes the loop; every pass updates the constitutional state, and the updated state grounds the next inference. Pinned as `DCIR_LOOP` in `types/dcir.ts` (order is constitutional data — sequencing corollary of Law XV; canary-guarded).

## 3. The three runtime domains and the twelve components

| Domain (`DCIR_RUNTIMES`) | Produces | State (honest, 2026-07-06) |
|---|---|---|
| **Conversational Runtime** | intent/reasoning → Intent Objects, Context Objects, Recommendations, Policies, Confidence | Partial — the copilot surfaces + ICE proposals exist (§8) |
| **Action Runtime** | deterministic execution → artefacts | Partial — receipted actions exist; artefacts don't re-enter context |
| **Observation Runtime** | everything becomes observable: conversation, tool outputs, documents, selection, editing, approval/rejection, undo, navigation, workflow progress, system events | **Partial (D1, 2026-07-06)** — event stream + observation seam live on the Dev Command Center (`services/dcir/eventStream.ts`); every other surface still unobserved |

The twelve architectural components: **Conversation Runtime**, **Inference Runtime**, **Observation Runtime**, **Action Runtime**, **Constitutional State Engine**, **Artefact Registry**, **Event Stream**, **Recommendation Engine**, **Affordance Generator**, **Invariant Intelligence Engine**, **Experience Graph**, **Feedback Coordinator**. None is greenfield-by-default: each is first inventoried against the existing organs (§8) and enters by composition (Extend, Don't Duplicate).

## 4. Constitutional State replaces conversation history

The reasoning substrate is a single evolving snapshot, not a transcript: **intent, goals, policies, constraints, active + previous artefacts, operator decisions, persona (T1 surface only), standing, preferences, confidence, task graph, experience graph, workflow** (`ConstitutionalStateSnapshot`, `types/dcir.ts`). D0 types the fields honestly loose — each hardens in the increment that implements its engine (D2), never before. Every conversational turn updates this state (`inv.interaction.112`); every recommendation is contextual to it, never to raw history (`inv.interaction.116`).

## 5. The event stream — the language of observation, tier-disciplined from birth

Everything emits events: `DocumentCreated`, `DocumentEdited`, `SelectionChanged`, `RecommendationAccepted`, `RecommendationRejected`, `ArtifactApproved`, `ArtifactRejected`, `UndoPerformed`, `NavigationOccurred`, `WorkflowAdvanced`, `ToolOutputProduced`, `ConversationTurn`, `PersonaChanged`, `SystemEvent` (`DcirEventKind`). An action without an event is invisible to the loop (`inv.interaction.113`).

**Identifier-tier discipline (Identity & Access Spine — non-negotiable):** T0 identifiers (`personaId`, `authProfileId`, `rootDid`) NEVER appear in an event payload. Every event carries a **T2-safe summary** (category labels + commitment refs, the activity-receipt `context_shared` discipline generalized) and a **tier marker** (`t1-browser-safe` | `t2-network-safe`); only T2 events are eligible for DVN anchoring. There is deliberately no T0 tier in the type — the contract makes the leak inexpressible, not merely forbidden.

## 6. Behavioural invariants — observed, never auto-canonical

Observation discovers **behavioural** invariants: observed constitutional patterns (e.g. "operator always edits before approving") — **NOT rules** (`inv.interaction.115`). They form a distinct class:

- Enter the substrate as status **`proposed`** — never `canonical` on entry.
- The `BehaviouralInvariant` type cannot express `canonical` at all: its status union is `'observed' | 'proposed'` by design.
- **Ratification stays with the operator** (`inv.cybernetics.111`: constitutional adaptation never bypasses ratification). The system may propose its own patterns; the operator canonizes them — through the existing invariant-substrate path under CCRL governance (CFS-019), never a parallel one.

## 7. Dynamic affordances — generated from intent, contained in capsules

UI affordances are generated dynamically from inferred intent — **recommendation engine, not toolbar** (`inv.interaction.117`). Every `Affordance` carries its constitutional basis (intent, artefact refs, workflow stage, T1 persona flags) and a **required capsule scope**: affordances emerge WITHIN the operator's active context, never as orphan output (Content Capsule Containment — the CLAUDE.md GOLDEN RULE applies to generated affordances exactly as to derivative content). An affordance with no scope must not render. DCIR is the universal primitive beneath aigentMe, Aigent Z, Studio Composer, Marketa, and every cartridge — only the visible actions change per surface.

## 9. Observe-mode-first rollout

DCIR ships observe-mode-first (the CFS-017 precedent, ratified): **observation instrumentation never blocks or mutates the surfaces it watches** until gating is separately ratified. D1's observation seam emits events and changes nothing else; renders are never blocked, actions never intercepted. Any future gate (an observation-informed deny, a generated affordance replacing a static one) is its own ratification, never a rider on the seam.

## D2 observation note — the loop improved itself (2026-07-06 live testing → 2026-07-07 fixes)

The operator's live-test report on the Dev Command Center (six findings: approval-blind proposal cards, weak left/right correlation, no flow-through on progression, a right pane that didn't follow free conversation, a gap-analysis extraction regression, and the absence of proactive turn progression — full record in CFS-015) **is itself the DCIR feedback loop operating**: observations of the running cognitive-action loop became the improvements to that loop within one day — Observation → Updated Context → Inference → Action, closed over the development process itself. Two DCIR-relevant deliveries rode the fix pass: (a) the approval flow-through's auto-close/auto-open transitions emit `devCapsuleClosedEvent`/`devCapsuleOpenedEvent` through the existing D1 helpers (observe-mode intact — the events narrate, they gate nothing); (b) the **first Feedback Coordinator slice (component #12)** shipped: an approval that advances the loop mints a one-shot `[observed]` auto-turn through the copilot's normal send path, so observation now INITIATES the next conversational turn rather than merely informing it. ## CDE increment — the DCIR becomes the Constitutional Development Environment (2026-07-07)

Operator + Aletheon directive (2026-07-07): when implementation is validated and a high / must-not-happen consequence comes back a FAILURE or PARTIAL failure, the loop must NOT accept it as "validated" and terminate — there must be an opportunity to remedy it IN the validation stage, plus a feedback loop for learning; and once code is proposed, only a PASSING consequence test unlocks deployment. This turns the Dev Command Center into the first **Constitutional Development Environment (CDE)**: development where a failed consequence check forks to remediation and deployment is gated on the consequence test.

Delivered this increment:

- **ICE-6 reframed → Constitutional Validation.** The `consequence_validation` stage keeps its enum key (back-compat — many maps key on it) but is displayed and framed as the constitutional consequence test.
- **ICE-7 Remediation** (`remediation`) and **ICE-8 Deployment Authorization** (`deployment_authorization`) inserted before `complete`. New STAGE_ORDER: intent_capture → context_assembly → gap_analysis → consequence_modeling → implementation → consequence_validation → remediation → deployment_authorization → complete. Remediation is a CONDITIONAL fork, not always traversed.
- **The validation→remediation fork** (`validationRequiresRemediation`, pure + canary-pinned): true when ANY high/critical consequence is `unintended`/`partial`, or the overall verdict is `fail`/`partial`. When true, an approved validation report advances to Remediation, not toward deployment. After remediation the loop re-validates (`revalidationRequired`) or proceeds when residual risk is explicitly accepted. `advanceStage` is fork-aware via `nextStage`; both stay pure.
- **Consequence-test-before-deploy** (`constitutionalThresholdMet`, pure): the validation report exists AND does not require remediation. Deployment Authorization's `canAdvance` and its Authorize affordance both gate on it — a failed consequence test cannot deploy.
- **Two new proposal kinds** — `remediation_plan` (`remedies[]` each with a `learningNote` — the captured lesson, the feedback-loop-for-learning the operator asked for; `residualRisk`; `revalidationRequired`) and `deployment_authorization` (`authorized`, `constitutionalThresholdMet`, `rationale`, `blockingConsequences`) — with schemas, apply logic committing to new `DevLoopState` fields (`remediationPlan`, `deploymentAuthorization`), and canary coverage.
- **Three Dev Receipts classes + the receipt-bug fix.** The receipt bug: `createDevLoopSession()` initialised `receipts: []` and nothing ever mutated it, so the Dev Receipts panel always read "No receipts yet." Every constitutional action now records its returned `receiptId` into `session.receipts` (`recordDevReceipt`, idempotent), typed `{ id, actionType, class, at }` and grouped by class: **Development** (`implementation_pack_generated`), **Constitutional** (`constitutional_validation_recorded`, `remediation_recorded`), **Deployment** (`deployment_proposed`, `deployment_authorized`).
- **Two new anchorable actionTypes** added to the receipt union and `ANCHORABLE_ACTION_TYPES` (strings-only, the one permitted DVN-pipeline change): `constitutional_validation_recorded`, `deployment_authorized` (plus `remediation_recorded`). New routes `POST /api/constitutional/validation-record` and `POST /api/constitutional/deployment-authorization` mirror the existing constitutional routes; summaries are T2-safe (goal excerpt + counts only, no T0 identifiers), and deployment-authorization refuses to record unless the caller asserts the threshold is met.

Honest about limits: the dev-loop session is still in-memory (no persistence, no cross-session state — **closed by the 2026-07-07 session-persistence increment below**); **execution stays human under CFS-016 D1** — Deployment Authorization records the authorization, it does not execute. The code runs in Claude Code and is pushed manually; the receipt is the provenance that the consequence test passed before deployment was authorized.

## CDE increment — dev-loop session persistence (2026-07-07)

Session persistence closes the CDE's longest-standing honest limit: `DevLoopState` lived only in React state and evaporated on refresh — intent, context pack, gap analysis, consequence canvas, implementation brief, validation report, remediation plan, deployment authorization, and the receipts list all lost with a keystroke. Delivered:

- **`dev_loop_sessions` table** (`supabase/migrations/20260707110000_dev_loop_sessions.sql`): `session_id` pk, `persona_id` (T0 — a DB ownership key ONLY, never serialized into responses), `stage`, the full `DevLoopState` as `state jsonb`, indexed on `(persona_id, updated_at DESC)`. RLS enabled with no policies — service-role access via the spine-gated route only (the `experiment_results` posture).
- **`/api/dev-command-center/sessions`** — GET returns the caller's most recent session (`?sessionId=` for a specific one); POST upserts keyed on `session_id`. Identity via `getActivePersona(request)` only; sessions are strictly caller-owned (an upsert against another persona's `session_id` is a 403; no admin bypass exists). Stage validation imports the now-exported `STAGE_ORDER` (`isDevLoopStage`) — no hardcoded stage set. The route T2-guards the jsonb: serialized state carrying `personaId` / `authProfileId` / `rootDid` / `fioHandle` / `kybeAttestation` keys is rejected (`findForbiddenStateKey`, pure + canary-pinned) — the dev-loop state must never carry T0 identifiers.
- **Hydrate-on-mount + debounced auto-save** in the Dev Command Center tab, both via `personaFetch` (spine rule): on arrival the tab restores the most recent session, primes the stage-transition observer (a restore is not a stage advance), and re-mounts the restored stage's capsule via `stageCapsuleId` so the surface lands where the operator left off. Hydration only ever lands on a still-pristine default session (`isPristineDevLoopSession`, pure) — it never clobbers work the operator started before the fetch returned. Every session mutation persists ~1.5s after it settles, fire-and-forget with `console.warn` on failure — persistence never blocks the loop. A subtle `saved` / `save failed` indicator rides next to the session id in the chrome. `localStorage` is deliberately unused for the session (server-first for critical state).
- **Round-trip canary**: `DevLoopState` is JSON-safe end-to-end — a stringify→parse round trip preserves `canAdvance` / `nextStage` / `advanceStage` semantics, including the validation→remediation fork; pinned in `tests/dev-command-center.test.ts` alongside the T2-guard predicate.

What remains honest limits: **multi-session browser UI** — the table already stores many sessions per persona and GET can fetch a specific one, but the tab surfaces only the most recent (a session picker is a later increment); **cross-device continuation UX** — state lives in the DB, so continuation works wherever the persona authenticates, but there is no explicit hand-off affordance yet.

## CDE DevTools scope decision (2026-07-07)

The operator asked whether the CDE DevTools viewport needs classic browser DevTools tabs (Console, Network, Elements). The ratified answer, recorded here as durable rationale so it is not relitigated:

- **(a) We do NOT rebuild browser DevTools — and why.** The real Chrome/Firefox F12 already provides Console, Network, Elements, and Sources for every client-side concern, better and free. Duplicating it would be an Extend-Don't-Duplicate violation and needless attack surface. The CDE DevTools viewport does NOT reimplement any of them.
- **(b) The non-redundant mandate is SERVER-SIDE visibility the browser cannot reach.** F12 sees the browser; it cannot see the server↔canister↔DVN pipeline or the platform escalation record. That gap is the CDE DevTools' unique job. Two observation surfaces were added: **platform telemetry** (recent DVN submissions with status + timing composed from the ops DVN probe, canister health + cycles, receipt pipeline counts by state, plus a best-effort per-instance request ring buffer) and the **escalation / platform log stream** (dvn_failed receipts surfaced read-only, newest first, each with the honest retry-route pointer).
- **(c) NO arbitrary-eval / script-execution console.** An eval console is exactly the CFS-016 D1 execution boundary the Terminal already enforces with its whitelist grammar. Scripted diagnostics grow the whitelisted Terminal (read-only, argument-validated, refused-by-default) — this increment added `dvn`, `logs`, `net`, `experiments`. Anything that would MUTATE goes through the consequence-test-before-deploy loop (Constitutional Validation → Remediation → Deployment Authorization), never a raw console. The Terminal remains a command surface, not a shell: no eval, no child_process, no dynamic dispatch beyond the whitelist map.
- **(d) A real CloudWatch log tail is deliberately deferred.** The AWS SDK is not a dependency and the "no new npm deps" constraint holds, so the escalation/log stream is sourced from DURABLE DB records (dvn_failed receipts) instead of a live server log tail — honest degradation, never fabricated log lines. Both the UI and the `logs` command label the source as DB-durable and name the CloudWatch tail as a follow-on gated on adding the SDK + IAM.
- **(e) Request telemetry is best-effort, per-instance, honestly labeled.** `services/devCommandCenter/requestTelemetry.ts` is a plain module-level ring buffer (cap 100). Lambda has no shared memory across compute instances and the buffer resets on cold start, so it reflects ONLY the current instance's traffic — a representative sample, never a complete log. It stores method + path TEMPLATE + status + latency + timestamp ONLY; query strings, bodies, headers, tokens, and T0 identifiers are structurally inexpressible and defensively stripped. The `net` command and the DevTools "Platform telemetry" section both carry the "this compute instance only — resets on cold start" label.

## D4 increment — the universal substrate hook (2026-07-07)

D4 begins: the observation seam, hand-wired identically on three surfaces through D1–D3, is extracted into ONE config-driven client hook so any surface adopts observation **by declaration** rather than by another hand-wired copy. This is Extend-Don't-Duplicate (CLAUDE.md) applied to interaction — the seam had reached the point where a fourth copy would be pure duplication.

Delivered this increment:

- **`services/dcir/useDcirSeam.ts`** — the universal substrate hook. `useDcirSeam({ surface, workflowStage?, activeCapsule? })` returns `{ events, observe, groundObservation }`. It composes the EXISTING D1/D2 organs (imports only — no forked logic): `observe` appends via `appendDcirEvent` (the session ring buffer, cap 50); `groundObservation` is memoized on `[events, surface, workflowStage, activeCapsule]` exactly as the hand-wired versions memoized. Observe-mode-only by construction: NO affordances, NO auto-act, NO persistence, NO cross-session memory live in the hook — those remain CONSUMERS of `events` on their surfaces.

- **The three-field server contract, now named and pinned.** `groundObservation` exposes EXACTLY `recentEvents` / `stateSnapshot` / `observedPatterns` — the keys the chat route's shared `pushDcirObservationLines` reads off each surface's ground context (`gc.recentEvents`, `gc.stateSnapshot`, `gc.observedPatterns`). Renaming any one is a breaking change across every adopting surface; the hook's header states this and `tests/dcir-substrate.test.ts` canaries it (exact-three-keys, ring-buffer cap, configured surface on the snapshot, T0-leak on the serialized ground observation).

- **Three surfaces adopted cleanly** — the proof that the pattern generalizes: the **Dev Command Center** (`DevCommandCenterTab.tsx`), the **aigentMe welcome split** (`AigentMeWelcomeSplitTab.tsx`), and the **Studio Composer** (`ComposerStudio.tsx`). Each swapped its hand-wired `[dcirEvents]` state + `observe` callback + three ground-context fields for `useDcirSeam(...)` + `...groundObservation`, behaviour-identical: every `observe(...)` call site is byte-for-byte unchanged, and every OTHER reader of the old `dcirEvents` (the DCC's `generateAffordances`/auto-act loop, aigentMe's intelligent-suggestion gate `suggestionLive`/`dcirEventCountRef`, aigentMe's Feedback Coordinator `autoPrompt`) now reads the SAME array via the hook's `events` (aliased locally to `dcirEvents` on those two surfaces to keep the downstream references untouched). The aigentMe Capsule↔Layout contract (CLAUDE.md PARAMOUNT) was NOT touched — no capsule/layout state, no `engageCapsuleAndMount`, no new effects, no layout resets; only the mechanical seam swap.

- **CCRL migrated to `useDcirSeam` (2026-07-09) — the last hand-wired seam is gone.** The CCRL research copilot (`CCRLResearchCopilotTab`) was the one surface deliberately left hand-wired when D4 landed, to bound that increment's blast radius. It now adopts the hook by declaration exactly as the other three did: its `[dcirEvents]`/`observe` block and its three inline ground-context fields (`compactDcirEvents` + `buildStateSnapshot` + `mineBehaviouralInvariants`/`compactBehaviouralInvariants`) were replaced by `useDcirSeam({ surface: SURFACE })` + `...groundObservation`. Behaviour-identical: every `observe(...)` call site is byte-for-byte unchanged, the snapshot stays surface-only (CCRL's stage/experiment were never threaded into the snapshot and still aren't — threading them is an optional micro-follow-on, not this swap), and the direct imports of the D1/D2 organs were dropped since the hook is now their single composition point. **All four originally-named DCIR surfaces (Dev Command Center, aigentMe, Studio Composer, CCRL) are now on the one substrate hook; hand-wiring a fresh seam anywhere is an Extend-Don't-Duplicate violation.** Remaining D4 frontier: adoption on surfaces beyond the four (Aigent Z, Marketa, per-cartridge) — each its own increment.

- **The rule for NEW surfaces:** adopt observation via `useDcirSeam`. Hand-wiring a fresh `[dcirEvents]` + `observe` + `{ recentEvents, stateSnapshot, observedPatterns }` block on a new surface is now an Extend-Don't-Duplicate violation — the substrate hook is the single authoritative home for the seam.

## Honest limits

- Invariants 112–118 are `proposed` in the seed crystal until the operator runs the ingest; this document is canon meanwhile (the CFS-018 discipline).
- The twelve components are named, not designed — each component's design is part of the increment that builds it.
- `ConstitutionalStateSnapshot`'s loose fields (`unknown`) are honest deferrals to D2, not extension points for ad-hoc payloads.
- Request telemetry is per-compute-instance and resets on cold start (Lambda has no shared memory); a complete request log is the deferred CloudWatch/durable-sink follow-on. The escalation log stream is DB-durable, not a raw server log tail — the CloudWatch tail is gated on the AWS SDK + IAM.
