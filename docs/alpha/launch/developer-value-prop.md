# AgentiQ Alpha — Developer Value Proposition

## Why build for AgentiQ OS instead of shipping a standalone tool?

A standalone tool lives or dies on its own distribution, trust model, and downstream usage. AgentiQ OS gives accepted supply a governed path into a live ecosystem.

When you build for AgentiQ OS, your contribution does not stop at publication. It can move through the Registry Ingestion Factory, become Registry-visible supply, be composed into structured experiences in Studio, get delivered through metaMe Runtime, generate participation and usage signal in live cartridges such as KNYT, and feed that signal back into what the ecosystem builds next.

In other words, AgentiQ OS offers more than packaging and hosting. It offers governed delivery, downstream visibility, and a path into a closed-loop supply system.

---

## What makes the governed model valuable

AgentiQ OS is not an open drop-box for unstructured plugins. It is a governed intake system.

That matters because governed intake makes accepted supply more valuable, not less:

- it improves composability by enforcing clearer interfaces and packaging standards
- it improves trust by making provenance, validation, and readiness legible
- it improves downstream usability because Studio and Runtime can work with supply that has already passed through the same intake logic
- it improves signal quality because usage happens inside a controlled loop rather than a disconnected marketplace

A contribution that passes governed intake is more likely to be discoverable, composable, and reusable across the ecosystem.

---

## The trust band system: a quality signal, not a gatekeeper

Every accepted contribution is assigned a trust band. In alpha, that should be understood as a readiness signal, not a social ranking system and not the final expression of the broader trust stack.

| Band | Meaning | Typical alpha implication |
|---|---|---|
| **L1 Experimental** | Early and unverified | Limited availability, used carefully |
| **L2 Verified Community** | Community-checked | Broader availability, stronger confidence |
| **L3 Production Candidate** | Meets a stronger production bar | Can be composed more confidently in Studio |
| **L4 Production Approved** | High-confidence production use | Strong placement in governed flows |
| **L5 Core Sovereign** | Platform-core quality | Reserved for highest-trust assets |

The trust band helps other parts of the system understand what your asset is ready for. It is not there to block good contributions from entering the ecosystem.

## How accepted supply flows through the ecosystem

The alpha loop for developers is:

```text
AgentiQ OS
  ↓ package + submit
Registry Ingestion Factory
  ↓ validate + assign trust band
Registry
  ↓ accepted supply becomes composable
Studio
  ↓ structured experiences are created
metaMe Runtime
  ↓ users participate in live worlds such as KNYT
PCS signal
  ↓ progression + feedback
future supply decisions
```

This matters because accepted supply is not treated as static inventory. It becomes part of a downstream experience loop where real user participation can surface what is useful, what is desired, and what should be improved or extended next.

## What developers can earn in alpha

The alpha uses a contained economic model to reward useful participation and supply without forcing a finalized platform-wide token system.

### $KNYT rewards for high-signal supply
Within the KNYT world, participation and contribution can generate $KNYT rewards. That creates the first live proving ground for cartridge-contained reward economics.

### PCS progression toward upstream contributor status
Developers are not only submitting assets. They are entering a progression system. The canonical PCS path for alpha is:

```text
participant → community → correspondent → operator → creator → upstream contributor
```

That means high-signal contribution does more than produce usage. It can move a contributor closer to deeper platform participation and more upstream influence over the ecosystem.

## What the SDK gives you

AgentiQ OS is already backed by a real SDK at:

```text
packages/agentiq-sdk/
```

The alpha developer surface includes:

- **`AgentIQClient`** — core client for talking to the platform
- **`A2AClient`** — typed agent-to-agent messaging support
- **typed messages and helper utilities** — structured calls and message construction
- **default personas** — including **Aigent C** as the onboarding and builder guide

This matters because the alpha is not asking developers to build blind. The SDK gives a real programmatic entry surface, and Aigent C gives a real guide for packaging and submission.

## What is in scope for alpha

AgentiQ OS accepts four submission categories during alpha:

- **ToolQube** — standalone tool or capability
- **SkillQube** — specialized workflow step or building block
- **WorkflowQube** — explicit multi-step orchestration
- **ConnectorQube** — integration or bridge to an external system

These map directly to the packaging and submission rules in the AgentiQ OS docs.

---

## What is not in scope for alpha

The alpha is intentionally narrower than the long-term platform vision.

Not in scope:

- the full public marketplace
- finalized ecosystem-wide tokenomics
- the full trust-scoring and risk/value research stack
- contributions that require changing platform-core code
- raw model weights without a usable wrapper
- proprietary bundles with unverifiable interfaces or unbounded exfiltration risk

The goal of alpha is to prove the governed loop, not to expose every future surface at once.

---

## Why the closed loop matters

Standalone tools often have to solve five hard problems on their own: distribution, trust, integration, user reach, and feedback. AgentiQ OS addresses those problems through the platform loop.

- **Distribution:** accepted supply enters a shared Registry instead of being buried in a personal repo
- **Trust:** validation and trust bands make readiness legible
- **Integration:** Studio can compose accepted supply into larger experiences
- **User reach:** Runtime and KNYT create real downstream exposure
- **Feedback:** participation and PCS signal show what is actually working

That is the core value proposition. You are not just shipping a tool. You are shipping into a governed ecosystem that can turn your work into real usage and structured signal.

## Alpha summary

AgentiQ OS is worth building for if you want more than a one-off deployment. It gives you a governed path from packaged contribution to composed experience to real downstream signal.

For alpha, the ask is simple:
- build one of the accepted contribution types
- package it clearly
- submit it through the Factory
- let the governed loop do the rest

That is what makes AgentiQ OS different from publishing a standalone tool in isolation.
