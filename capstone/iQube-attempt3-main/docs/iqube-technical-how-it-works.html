iQube — Technical overview

How the React app, encryption layer, decentralized storage, Polygon Amoy, Supabase, and the local Node server fit together in this repository (as implemented today).

Open this file in a browser for the layout. Contract address and RPC URLs come from src/utilities/constants.ts and env vars — update this page if you redeploy.

What you are building

An iQube is an ERC-721 NFT on Polygon Amoy whose tokenURI points to a JSON document (on Pinata/IPFS or Autonomys Auto-Drive). That JSON holds public-style attributes and, when the user added private fields, a ciphertext blob (AES-256-GCM). The data encryption key (DEK) is not stored on-chain in the current Solidity contract; it is stored through a Supabase Edge Function (iqube-key-service) after the user signs an authorization message with their wallet.

Mint pipeline (wizard)

1

Collect metadata

User picks an iQube type (Data, Content, Tool, Model, Agent), category, access policy, and optional private fields. Types and scoring live in src/types/iqube.ts.

Browser
2

Encrypt private payload (if any)

EncryptionModule in src/utilities/encryption.ts uses the Web Crypto API: AES-256-GCM, random IV, ciphertext + auth tag. The exported hex DEK stays in memory until step 6.

The wizard embeds iv, authTag, and encryptedData (hex) inside the NFT metadata JSON under attributes (trait blakQube).

AES-GCMClient-side
3

Upload JSON metadata

Pinata: pinata.upload.json via src/utilities/pinata-config.ts; gateway URL from VITE_GATEWAY_URL.

Auto-Drive: POST http://localhost:4000/api/autodrive-upload on autodrive-server.cjs, which uses @autonomys/auto-drive. Returns a public url stored as the on-chain URI.

IPFS / Auto-DriveLocal server
4

Mint ERC-721

useMintQube → mintQube(to, metaQubeLocation) in contractUtils.ts using viem + injected wallet. Chain: polygonAmoy. ABI: src/utilities/iqubeAbi.ts.

On-chain function (matches contract/iqube.sol): mintQube(address to, string uri) — stores the metadata URL and minter; no encryption key in calldata.

Polygon Amoyviem
5

Post-mint bookkeeping

getTokenIdFromMintReceipt parses the Transfer log to get tokenId.

If Supabase is configured: insert into iqubes (title, type, IPFS URL/hash, tx hash, policies, etc.). For accessPolicy === "specific", rows go to iqube_access_list.

Supabase DB
6

Store DEK (encrypted iQubes only)

authorizeDekStorage in keyWrapping.ts asks the wallet for personal_sign on a structured message, then invokes the Edge Function iqube-key-service to persist the wrapped key server-side. If Supabase is missing, mint is blocked when there are private fields.

Edge FunctionWallet signature

Decrypt pipeline (/decrypt)

  1. Resolve ownerOf(tokenId); if not owner, allow decrypt only if the wallet appears in iqube_access_list (Supabase).
  2. Read getMetaQubeLocation(tokenId) from the contract, then fetch the JSON. Autonomys gateway URLs can be proxied through localhost:4000/api/autodrive-metadata to avoid browser CORS issues.
  3. Call requestDekForDecryption (Edge Function + signature) to obtain the DEK; support unwrapLegacyDek for older records if needed.
  4. EncryptionModule.Decrypt rebuilds the GCM ciphertext and returns plaintext fields.

On-chain contract surface

The frontend ABI includes: mintQube, getMetaQubeLocation, minterOf, ownerOf, transferQube. OpenZeppelin ERC721 behavior applies for approvals and tokenURI. Source: contract/iqube.sol.

What each layer stores

Polygon Amoy

Ledger
  • NFT ownership and token IDs
  • Metadata URI per token (IPFS gateway or Auto-Drive URL)
  • minterOf mapping

Pinata / Auto-Drive

Content-addressed JSON
  • Public attributes + encrypted blakQube blob
  • No DEK in the JSON (only ciphertext components)

Supabase

Index + key custody
  • iqubes rows for app features
  • iqube_access_list for shared decrypt
  • Edge Function stores / releases DEKs after wallet proof

Browser

UX + crypto
  • Wizard, wallet connect, signing
  • AES-GCM encrypt/decrypt
  • Never persists DEK to disk by default

Local Node server

autodrive-server.cjs
  • Uploads metadata to Auto-Drive; proxies metadata fetch for decrypt
  • Optional agent chat + rate limiting when Ollama / env vars are set
Legacy / alternate paths in the repo: MetaContract.ts and AvalancheMetaInterface.ts target older Web3 flows (e.g. mintQube(uri, encryptionKey) with three conceptual arguments). The current UI mint path uses contractUtils.ts + iqubeAbi.ts (two-arg mint). Older HTML docs under docs/ may describe the three-arg design; trust the Solidity + ABI in this repo for the deployed shape you are actually calling.
