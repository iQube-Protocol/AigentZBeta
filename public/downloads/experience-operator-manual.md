# Paper 1 — The metaMe Operator's Manual

*Runtime, Studio, Registry, nanOS, Cartridges, Codexes, and Sovereign Experience Operations.*

## 1. Opening orientation

metaMe is the sovereign experience runtime for users, agents, content, workflows, identity, and digital services.

It is not simply another application layer. It is the operating environment where a user defines what enters their experience, how it is governed, which agents may act, what information may be used, what priorities matter, and what outcomes the system should help them achieve.

The metaMe suite consists of three primary operating surfaces:

- **metaMe Runtime** — the user's live sovereign experience space.
- **metaMe Studio** — the professional-grade composition layer for creating and publishing experiences.
- **iQube Registry** — the canonical index and first production-grade filter for iQubes and agent-legible primitives.

Between the Registry and Studio sits **nanOS**, the integrated operating and filtering layer that prepares Registry assets for Studio-grade and Runtime-grade use.

The open creation and interoperability layer beneath this is **AgentiQ OS**, which gives developers, agents, and technical operators a way to create complex primitives, tools, workflows, agent services, and cartridge-compatible modules.

The operating stack is:

```
AgentiQ OS
  ↓
iQube Registry
  ↓
nanOS
  ↓
metaMe Studio
  ↓
metaMe Runtime
  ↓
metaMe Experience Catalogue / User Runtime Activation
```

- AgentiQ OS opens the pipeline.
- The iQube Registry canonizes and indexes valid primitives.
- nanOS filters and operationalizes Registry assets for Studio and Runtime.
- metaMe Studio composes professional-grade experiences.
- metaMe Runtime activates those experiences for users under policy, permission, and context.

---

## 2. The core operating doctrine

metaMe is built around a simple operating doctrine:

> **The user should govern their experience, not be governed by platforms.**

In legacy software, the application decides the interface, the workflow, the data structure, the content hierarchy, and the operating assumptions. The user adapts to the application.

In metaMe, the user defines an experience model, activates Cartridges, governs agent permissions, and uses aigentMe to coordinate experiences across content, services, identity, time, goals, and workflows.

The system does not ask non-technical users to think like software architects. Instead, it lets users and operators express domain expertise, desired outcomes, policies, and experience intent. The system then helps translate those intentions into iQubes, Capsules, Pills, Smart Content, workflows, agent services, and Runtime surfaces.

> The user provides domain expertise and desired experience clarity. metaMe handles the technical translation, policy recommendations, iQube selection, surface composition, and activation pathway.

## 3. The operating stack

### 3.1 AgentiQ OS

AgentiQ OS is the open-source creation and interoperability layer.

It supports developers, agents, and technical teams creating:

- complex iQube primitives
- ToolQubes
- ModelQubes
- AigentQubes
- agentic workflows
- cartridge-compatible modules
- MCP/App integrations
- service components
- policy-aware primitives
- code-defined experience components

Simple iQubes such as DataQubes and ContentQubes can be created inside Studio or Runtime without code. More complex primitives often originate through AgentiQ OS.

AgentiQ OS is the open pipeline into the metaMe ecosystem.

### 3.2 iQube Registry

The iQube Registry is the canonical index and first production-grade filter for iQube primitives.

All iQube-grade primitives that enter the broader metaMe ecosystem pass through the Registry hurdle. The Registry establishes whether a primitive is valid, discoverable, policy-aware, versioned, agent-legible, and eligible for downstream use.

The Registry answers:

- What is this iQube?
- What type of primitive is it?
- Who created it?
- What is public?
- What is private?
- What policies govern use?
- What agents may read or invoke it?
- What access rights apply?
- What receipts or provenance records support it?
- What surfaces may use it?
- Is it eligible for Studio?
- Is it eligible for Runtime?

The Registry is not merely a database. It is the canonical reference layer for iQubed assets.

### 3.3 nanOS

nanOS sits between the iQube Registry and metaMe Studio.

It is the integrated operating and filtering layer that prepares Registry assets for Studio-grade and Runtime-grade use.

nanOS operationalizes:

- iQubes
- Personas
- Cartridges
- internet-cartridge operations
- SmartTriad relationships
- Smart Content
- Liquid UI
- Smart Menu
- Smart Wallets
- CoPilots
- DVN receipts
- policy controls
- access gating
- disclosure rules
- agent permissions
- Runtime compatibility

The metaMe Client Protocol describes how Personas, internet-cartridge operations, and iQubes work together. nanOS is the integrated surface that makes these components operational as a coherent, policy-governed foundation for the metaMe suite.

- AgentiQ OS provides open interoperability.
- The Registry establishes canonical primitive status.
- nanOS filters, integrates, and prepares those primitives for Studio and Runtime.

### 3.4 metaMe Studio

metaMe Studio is the professional-grade composition layer.

It is where operators, creators, teams, agencies, and domain experts compose experiences using registered primitives, iQubes, Smart Content, Codex-grade assets, workflows, agents, Liquid UI surfaces, and cartridge structures.

Studio is where experiences are shaped before they reach general user Runtime contexts.

Studio supports:

- experience composition
- Cartridge creation

- Pill creation
- Codex-grade asset preparation
- Smart Content packaging
- workflow commissioning
- missing-skill tendering
- policy review
- preview and testing

### 3.5 metaMe Runtime

metaMe Runtime is the user's sovereign experience environment.

Runtime is where the user activates Cartridges, interacts with Capsules and Pills, receives Next Best Experience recommendations, governs agents, configures their experience model, and uses aigentMe to align personal, professional, creative, and ecosystem goals.

Runtime is the highest consumer-grade filter.

Anything generally available in the metaMe Runtime must be safe, legible, useful, policy-compatible, and appropriate for consumer-grade activation.

### 3.6 metaMe Experience Catalogue

The metaMe Experience Catalogue is the general-consumption catalogue for approved Runtime-ready experiences.

It is similar to an App Store, but for sovereign experiences.

The Catalogue provides consumer protection, quality controls, policy review, provenance checks, and Runtime safety standards for:

## 7. Codex as surface and grade

Codex has two meanings in metaMe. It is both **a surface** and **a grade of content**.

### 7.1 Codex as surface

A Codex surface is the canonical ledger or knowledge space inside a Cartridge.

It contains official, trusted, canonical, receipt-backed, or iQubed material.

A Codex may include:

- canonized papers
- official articles
- protocol documents
- scripts
- lore
- images
- media assets
- provenance records
- collectible references
- registered ContentQubes
- DVN-backed records
- canonical workflows
- official service definitions

### 7.2 Codex as grade

Codex-grade content is canonical-grade content.

A Codex asset may be surfaced outside a Codex tab while remaining part of the Cartridge's canonical ledger.

For example, a Codex-grade KNYT asset may appear:

- inside the KNYT Codex
- on the KNYT Cartridge home surface
- inside a Capsule
- inside a Runtime stream
- inside a Pill
- inside a user's aigentMe-governed Runtime recommendation
- inside a navigation flow
- inside a quest or task surface

Its location may change, but its canonical status remains.

> Codex is the canonical ledger. Codex-grade is the canonical quality of the asset. Codex-grade assets can travel across surfaces without losing their ledger status.

It is configured by the user and their aigentMe inside the metaMe Cartridge.

The Experience Model consists of:

- Experience Strategy
- Experience Ladder
- Experience Matrix
- Experience Journey
- Experience State

### 10.1 Experience Strategy

The Experience Strategy defines what the user is trying to accomplish.

It captures: user goals, operating intent, current priorities, time horizons, constraints, desired outcomes, sovereignty preferences, business or personal focus, what should be protected, what should be amplified.

### 10.2 Experience Ladder

The Experience Ladder defines progression. It can describe how a user, creator, venture, Cartridge, or community member moves from one stage to the next.

```
Consumer → Curator → Remixer → Creator → Developer → Correspondent
```

*Or:*


*For example:* development maturity, commercialization maturity, personal priority, business priority, time sensitivity, repair risk, opportunity value, sovereignty gain, ecosystem value.

### 10.4 Experience Journey

The Experience Journey describes the sequence of meaningful steps. It may include onboarding, activation, learning, creation, contribution, publishing, feedback, rewards, governance, next best action, next best experience.

### 10.5 Experience State

Experience State is the live operating condition of the user's Runtime.

It tracks: active Cartridges, active Capsules, active tasks, active policies, current goals, recent activations, open approvals, agent permissions, deferred recommendations, cartridge priorities, cross-cartridge conflicts, repair risks, next best actions.

> The Experience Model tells the Runtime what matters. aigentMe uses the Experience Model to guide what appears, what waits, what requires approval, and what should happen next.

## 13. SmartTriad operations

SmartTriad is the operational triangle of:

- CoPilots
- Codexes
- Wallets

The relationship among these is typically defined within a Cartridge. Once defined, the SmartTriad relationship follows the Cartridge's assets wherever they are surfaced.

For example, the KnOw1 KNYT agent may be available to users who interact with a KNYT Capsule in their aigentMe-governed Runtime, because that Capsule belongs to the KNYT Cartridge's SmartTriad relationship.

For alpha release, these relationships are confined to Cartridge activation to minimize complexity. The KnOw1 agent is available when the KNYT Cartridge is activated.

Over time, this becomes more component-level and fluid. A Codex-grade asset, Capsule, or Pill may carry its SmartTriad relationship with it wherever it appears, making the right CoPilot, Codex context, and Wallet functionality available without forcing the user to manage the technical complexity.

### 13.1 CoPilots

CoPilots guide users through experiences. They explain context, answer questions, recommend actions, interpret Cartridge logic, help configure policy, assist publishing, prepare approvals, coordinate workflows, hand off to specialist agents.

- **aigentMe** is the user's primary Runtime CoPilot.
- Cartridge-specific CoPilots, such as **KnOw1** in KNYT, provide domain-specific guidance.

### 13.2 Codexes

Codexes ground experiences in canonical knowledge. They preserve trusted context, hold canonical-grade assets, connect assets to receipts, support search and navigation, distinguish drafts from canon, provide agents with reliable reference material.

### 13.3 Wallets

Smart Wallets authorize, gate, pay, prove, and record. They support access gating, payment gating, token gating, proof of provenance, signing, rewards, receipts, ownership references, QriptoCENT settlement, $KNYT or cartridge-native economies, identity-linked permissions.

> The CoPilot guides. The Codex grounds. The Wallet authorizes.

## 15. iQube Registry and agent legibility

The iQube Registry is the canonical source for iQube primitives.

The Registry section should include the IANA-inspired iQube Card standard developed for agent legibility.

> The Registry is the canonical source. The iQube Card is the agent-readable calling card.

The iQube Card allows agents to safely understand:

- what an iQube is
- what type it is
- what actions are permitted
- what policies apply
- what access rules govern it
- what surfaces it supports
- what Cartridge or Codex it belongs to
- how it may be requested, invoked, or used

The IANA-inspired discovery pattern may include endpoints such as:

- `/.well-known/iqube-catalog`
- `/.well-known/iqube-manifest.json`
- `/api/iqubes/:id/card`
- `/api/iqubes/:id/policy`
- `/api/iqubes/:id/actions`

This makes iQubes more legible to agents without exposing private payloads or bypassing policy.

## 17. Cartridge creation playbook

### Step 1: Define the Cartridge purpose

The operator defines: domain, audience, purpose, desired outcomes, operator identity, whether the Cartridge is personal, private, partner, public, or commercial.

### Step 2: Define Cartridge policy

The operator configures: who can access the Cartridge, who can create inside it, who can submit content, what requires review, what can be shared, what can become Codex-grade, what identity state is required, what agents can operate inside it, what rewards or rights apply.

### Step 3: Commission primitives

The system helps commission: required iQubes, Codex structure, initial Tabs, Active Surfaces, Capsules, Pills, Smart Content, Smart Wallet hooks, CoPilot roles, workflow modules, permissions.

### Step 4: Build in myCanvas, Runtime, or Studio

The operator can privately compose and preview the Cartridge.

### Step 5: Invite testers

The operator can invite others to the Cartridge on a limited basis for testing and feedback.

### Step 6: Submit for Registry review

If the Cartridge is intended to enter the metaMe ecosystem, its primitives must be Registry-legible and policy-compliant.

### Step 7: Submit for Studio or Runtime review

If the Cartridge is intended for broader use, it must pass Studio-grade and Runtime-grade filters as appropriate.

### Step 8: Publish to the Experience Catalogue

Approved Cartridges can enter the metaMe Experience Catalogue for general activation.

# Paper 2 — The Experience Vibing Operator's Manual

*How non-technical professionals create sovereign agentic experiences in metaMe.*

## 1. Why Experience Vibing exists

Experience Vibing makes AI programming available to non-technical professionals.

It allows domain experts, creators, operators, strategists, educators, media teams, venture builders, and service providers to create sophisticated agentic experiences without needing to think like software engineers.

The operator does not need to manually manage:

- schemas
- iQube types
- access-control graphs
- agent invocation formats
- policy syntax
- UI rendering
- Registry metadata
- DVN receipt structure
- runtime compatibility
- code-level workflow orchestration

Instead, the operator focuses on:

- domain expertise
- audience needs
- desired outcomes
- experience clarity
- policy intent
- trust boundaries
- content quality
- business logic
- user journey
- what should happen next

The system translates that intent into experience architecture.

> Experience Vibing is not vibe coding for apps. It is **outcome-led experience composition for sovereign Runtime environments.**

---

## 2. What Experience Vibing is

Experience Vibing is the process of creating, shaping, commissioning, activating, testing, publishing, and improving experiences through metaMe.

It uses: user intent, Experience Models, Cartridges, iQubes, Smart Content, Studio Composer, SmartTriad relationships, Liquid UI, Runtime policy, aigentMe guidance, Registry filters, Studio filters, Runtime filters.

The operator works in natural, domain-specific terms. The system recommends:

- which iQubes are needed
- what policies should apply
- what access controls are appropriate
- whether the experience should be a Capsule, Pill, Tab, Cartridge, or Codex-grade asset
- whether a missing workflow or skill needs to be commissioned
- whether the asset is ready for Registry
- whether it needs Studio composition
- whether it is safe for Runtime
- whether it can be published to the Experience Catalogue

The operator remains in the loop and authorizes recommendations.

---

## 3. The Experience Vibing loop

The core loop is:

1. Declare intent
2. Define the desired experience outcome
3. Select or create Cartridge context
4. Commission required iQubes, assets, skills, or workflows
5. Review system recommendations
6. Authorize policy and access settings
7. Compose in myCanvas, Runtime, Cartridge, or Studio
8. Package as Capsule, Pill, Tab, Codex-grade asset, or Cartridge
9. Preview privately
10. Invite testers or collaborators
11. Publish privately, socially, or to a Cartridge
12. Submit to Registry, Studio, and Runtime filters as appropriate
13. Activate in Runtime or publish to the Experience Catalogue
14. Collect feedback, receipts, and state changes
15. Refine, version, canonize, archive, or expand

This loop lets non-technical operators move from intent to working agentic experience without carrying technical cognitive load.

---

## 4. The role of commissioning

Commissioning is a core part of Experience Vibing.

The operator may not know which iQubes, workflows, skills, agents, or integrations are needed. They do not need to know.

They describe the desired experience. The system identifies what must be selected, created, commissioned, or reviewed.

Commissioning may include:

- selecting existing iQubes
- creating new ContentQubes or DataQubes
- requesting a ToolQube
- requesting an agent workflow
- requesting a missing integration
- asking Studio Composer to generate a mini-RFP
- tendering a capability to dev teams or code agents
- defining acceptance criteria
- preserving client confidentiality
- managing IP rights

This is especially important when a user wants an experience that does not yet exist.

*For example:* a media operator wants a Cartridge that turns Qriptopian articles into social Capsules, video prompts, community quests, and email sequences.

If the needed workflow exists, Studio Composer can assemble it. If it does not exist, Studio Composer can create a mini-RFP for the missing workflow while protecting confidential details and specifying IP rights, delivery criteria, policy requirements, and Registry packaging expectations.

*Examples:* Order Tab, Publishing Tab, Quest Tab, Runtime Policy Tab, Studio Review Tab, Codex Tab.

The user needs to do something.

*Examples:* submit, activate, configure, claim, join, complete, approve, publish, invite.

The experience is a complete compound unit.

*Examples:* onboarding experience, quest, challenge, media package, campaign module, partner proposal, learning sequence, living canon contribution.

The experience is compact and action-oriented.

*Examples:* "Review this", "Claim this", "Complete setup", "Read this", "Approve policy", "Invite tester".

*Examples:* Codex Grade, Needs Review, Runtime Approved, Private, Canonized, Quest Available.

*Examples:* Activate, Publish, Submit, Claim, Invite, Approve, Remix, Open.

### 7.1 Create in myCanvas

A user may create privately in myCanvas. They can draft, preview, revise, activate privately, invite testers, prepare for Registry submission, keep the work private.

A user may create inside their Runtime. They can publish to myCanvas, private Runtime, invited collaborators, third-party social platforms, or a third-party Cartridge (subject to that Cartridge's policies).

A user may create within a Cartridge. That creation is governed by the Cartridge's publishing policy.

A user or operator may create in Studio when professional-grade composition is required. Studio supports multi-surface composition, workflow assembly, agent coordination, policy review, iQube preparation, preview, versioning, Studio-grade packaging, Runtime submission.

### 7.5 Publish to the metaMe ecosystem

To publish into the broader metaMe ecosystem, an asset must pass the relevant filters. The standard pathway is:

```
Creation
  ↓
iQube Registry
  ↓
nanOS filtering
  ↓
Studio review, if needed
  ↓
Runtime review, if needed
  ↓
metaMe Experience Catalogue
```

## 8. Experience Vibing examples
